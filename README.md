# LLM 모델 비교·선정 보고서

## 1. 비교 목적

본 보고서는 동일한 업무 과업인 **AI 뉴스 큐레이터(뉴스 자동화)**를 수행하기 위해 GPT 5.5, Claude Sonnet 4.6, Gemini 2.5 Flash를 대상으로 성능을 비교하고, 뉴스 자동화 업무에 가장 적합한 LLM을 선정하는 것을 목적으로 한다.

비교는 동일한 테스트 입력을 사용하여 수행하였으며, 뉴스 선별 정확성, 사실성, 형식 준수, 문맥 유지 등 실무 활용에 필요한 항목을 기준으로 평가하였다.

# 2. 업무 과업

## 업무명

**AI 뉴스 큐레이터 (News Automation AI)**

## 업무 내용

여러 뉴스 기사 중 국민 생활에 영향을 주는 핵심 뉴스를 선별하고, 중복 기사를 제거한 뒤 객관적인 요약을 생성한다.

출력은 다음 형식을 따른다.

* 선정 이유
* 핵심 요약
* 국민에게 미치는 영향
* 확인 필요

# 7. 모델별 분석

## GPT 5.5

GPT 5.5는 뉴스 자동화 업무에서 가장 안정적인 결과를 생성하였다.

특히 출력 형식 준수, 문맥 유지, 시스템 프롬프트 반영 능력이 우수하였으며, 기사에 없는 내용을 생성하지 않고 "확인 필요"를 적절하게 표시하였다. 또한 시스템 프롬프트, Few-shot 예시, 환각 테스트, 실행 로그까지 일관성 있게 설계할 수 있어 실무 적용성이 높았다.

---

## Claude Sonnet 4.6

Claude Sonnet 4.6은 프롬프트 개선 방향을 분석하는 능력이 뛰어났다.

확인 질문 조건을 실제 예외 상황 중심으로 축소해야 한다는 점, 뉴스 중요도 기준에 우선순위를 부여해야 한다는 점, Markdown 또는 JSON 구조화를 통해 자동화 파이프라인과의 연동성을 높일 수 있다는 점을 제안하였다.

환각 테스트에서는 대부분의 항목을 통과하였으나, 제목만으로 동일 사건 가능성을 추정하는 부분은 조건부 PASS로 평가하였다.

---

## Gemini 2.5 Flash

Gemini 2.5 Flash는 프롬프트 개선 과정을 체계적으로 분석하였다.

단계적 접근(Reasoning Prompt), 확인 질문, 출력 구조화, 자동화 친화적인 설계가 뉴스 자동화 업무에 효과적이라는 점을 제시하였다. 또한 입력 검증 강화와 Markdown 구조화를 통해 API 및 자동화 시스템과의 연계성을 높일 수 있다는 개선 방향을 제안하였다.

---

# 8. 프롬프트 개선 과정

본 과제에서는 하나의 모델만 사용하지 않고 **GPT 5.5를 중심으로 Claude Sonnet 4.6과 Gemini 2.5 Flash의 피드백을 반복적으로 반영**하는 방식으로 프롬프트를 개선하였다.

초기 v1 프롬프트는 뉴스를 단순 요약하는 수준이었으나, v2에서는 단계적 접근, 확인 질문, 사실 검증 규칙을 추가하였다. 이후 Claude Sonnet 4.6의 피드백을 반영하여 중요도 우선순위, 예외 처리, 출력 구조를 개선하였고, Gemini 2.5 Flash의 피드백을 반영하여 입력 검증 강화와 자동화 친화적인 Markdown 구조를 적용하였다.

이와 같은 반복 개선(Iteration)을 통해 단일 모델만 사용할 때보다 프롬프트의 안정성과 재현성을 높일 수 있었다.

---

# 9. 최종 선정 모델

최종 모델은 **GPT 5.5**로 선정하였다.

선정 이유는 다음과 같다.

1. 뉴스 자동화 업무에서 가장 높은 정확도와 안정성을 보였다.
2. 긴 대화에서도 조건 변경과 출력 형식 변경을 일관되게 유지하여 문맥 유지 능력이 가장 우수하였다.
3. 기사에 없는 내용을 생성하지 않고 "확인 필요"를 표시하여 환각을 효과적으로 방지하였다.
4. 시스템 프롬프트, Few-shot, 환각 테스트, 실행 로그를 포함한 전체 문서를 가장 일관성 있게 생성하였다.
5. Claude Sonnet 4.6과 Gemini 2.5 Flash의 피드백을 반영하여 프롬프트를 반복 개선함으로써 뉴스 자동화 업무에 적합한 최종 프롬프트를 완성할 수 있었다.

## 2. 문제 정의

현대 사회에서는 하루에도 많은 뉴스가 생산된다. 사용자가 모든 기사를 직접 읽고 중요한 뉴스를 선별하기에는 시간과 노력이 많이 필요하다.

또한 같은 사건을 다룬 중복 기사, 자극적인 제목, 사실 확인이 부족한 정보가 섞여 있어 핵심 이슈를 빠르게 파악하기 어렵다.

따라서 본 시스템은 다음 문제를 해결하는 것을 목표로 한다.

* 중요한 뉴스와 중요하지 않은 뉴스를 구분하기 어렵다.
* 동일 사건에 대한 중복 기사가 많다.
* 기사 내용에 없는 추측성 요약이 발생할 수 있다.
* 뉴스 요약 형식이 일정하지 않으면 자동화 시스템에 연결하기 어렵다.
* 사실, 정책, 수치가 포함된 정보는 검증 기준이 필요하다.

## 4. 입력 데이터

AI 뉴스 큐레이터는 다음과 같은 데이터를 입력받는다.

| 입력 항목  | 설명                         |

| 언론사명   | 기사 출처 확인용 정보               |

입력 데이터는 RSS Feed, 뉴스 API, 웹 크롤링 결과, 사용자가 직접 입력한 기사 목록 등으로 제공될 수 있다.

## 5. 출력 형식

최종 출력은 Markdown 형식을 사용한다.

```markdown
### 기사 1
- **제목**:
- **선정 이유**:
- **핵심 요약**:
  1.
  2.
  3.
- **국민에게 미치는 영향**:
- **확인 필요**:
```

### 출력 규칙

* 가장 중요한 뉴스 3건을 선정한다.
* 입력 기사가 3개 미만이면 존재하는 기사 수만큼만 출력한다.
* 부족한 기사를 임의로 생성하지 않는다.
* 기사에 없는 내용은 작성하지 않는다.
* 불확실한 정보는 “확인 필요”로 표시한다.
* 동일 사건의 중복 기사는 하나로 통합한다.

## 8. 시스템 프롬프트

```text
당신의 이름은 NOVA입니다.

당신은 객관적인 뉴스 큐레이터이자 팩트체크 에디터입니다.

당신의 목표는 여러 뉴스 기사 중 국민에게 중요한 뉴스를 선별하고, 사실에 근거한 요약을 제공하며, 검증되지 않은 내용을 생성하지 않는 것입니다.

당신은 언론인의 의견을 작성하는 AI가 아니라, 객관적인 뉴스 편집자 역할을 수행합니다.

다음 순서로 업무를 수행합니다.

1단계. 입력 데이터 확인

다음 조건에 해당하면 최종 결과를 생성하지 말고 최대 3개의 확인 질문만 출력합니다.

- 기사 본문이 없는 경우
- 동일 사건 여부를 판단하기 어려운 경우
- 기사 내용이 불충분하여 사실 검증이 어려운 경우

2단계. 뉴스 선별
```

# 15. 심층 인터뷰 및 기술 설명

## 15.1 환각(Hallucination)과 창의적 생성의 차이

본 과제에서 환각(Hallucination)은 **사실, 수치, 정책, 계산 등 검증 가능한 정보를 근거 없이 생성하거나 틀린 내용을 사실처럼 제시하는 현상**으로 정의하였다.

반면 창의적 생성은 소설, 광고 카피, 아이디어 제안처럼 정답이 존재하지 않는 영역에서 새로운 내용을 생성하는 것으로, 허구임을 명확히 밝히거나 세계관의 일관성을 유지한다면 환각으로 판단하지 않는다.

따라서 뉴스 자동화 업무는 사실 기반 업무이므로 기사에 없는 내용을 추측하거나 수치를 임의 생성하는 경우를 환각으로 간주하였다.

---

## 15.2 무료 모델만 사용할 경우 품질 유지 전략

비용 제약으로 무료 모델만 사용해야 하는 경우에는 모델 성능보다 **프롬프트 구조와 검증 절차를 강화**하여 품질을 유지한다.

적용 전략은 다음과 같다.

* 입력 템플릿을 표준화하여 누락 정보를 최소화한다.
* 단계적 프롬프트(v2)를 사용하여 확인 질문 → 기사 검토 → 요약 순으로 처리한다.
* 기사에 없는 내용은 생성하지 않고 "확인 필요"로 표시한다.
* GPT, Claude, Gemini 무료 버전을 교차 검증하여 결과를 비교한 후 최종 결과를 선정한다.

이와 같은 절차를 통해 무료 모델 환경에서도 안정적인 품질을 확보할 수 있다.

---

## 15.3 정책이나 규정이 자주 변경되는 경우 환각 감소 전략

뉴스 자동화 업무에서는 정책, 법령, 수치가 자주 변경될 수 있으므로 다음과 같은 운영 전략을 적용하였다.

* 기사 본문에 없는 내용은 생성하지 않는다.
* 최신 기사 또는 공식 자료를 우선 근거로 사용한다.
* 확인이 어려운 내용은 "확인 필요" 태그를 출력한다.
* 정책 변경 시 시스템 프롬프트보다 입력 데이터(RSS, 기사 본문)를 우선 적용한다.
* 오래된 예시(Few-shot)가 최신 정책과 충돌하면 최신 입력을 우선한다.

이를 통해 오래된 학습 정보로 인한 환각을 최소화하였다.

---

## 15.4 장기 대화에서 문맥이 끊기는 문제 해결 전략

10턴 이상의 대화에서는 조건이 누적되면서 일부 요구사항이 누락될 수 있다.

이를 해결하기 위해 프롬프트를 다음과 같이 개선하였다.

1. 대화 시작 시 현재 적용 중인 조건을 내부적으로 유지한다.
2. 사용자가 조건을 변경하면 변경된 부분만 수정하고 나머지 조건은 유지한다.
3. 출력 형식 변경 시 뉴스 선정 기준은 그대로 유지한다.
4. 필요하면 현재 적용 중인 조건을 다시 요약하여 확인한다.
5. 최종 출력 전 기존 요구사항(톤, 대상, 금지사항)을 다시 검토한 후 응답한다.

이 구조를 적용하여 조건 변경, 출력 형식 변경, 톤 변경이 발생해도 문맥을 안정적으로 유지할 수 있었다.
